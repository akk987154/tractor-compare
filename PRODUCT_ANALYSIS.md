# 🚜 TractorCompare — 产品经理分析报告

> 分析日期：2026-06-12 | 版本：v0.4 | 分析维度：产品策略 / 用户体验 / 数据策略 / 增长 / 技术架构

---

## 一、产品概览

TractorCompare 是一个面向中国农机购买者的拖拉机规格对比工具。收录 17 款真实拖拉机（5 大品牌），提供并排参数对比、雷达图可视化和智能胜出判定。

**核心价值主张**：帮助农户在购买拖拉机时做出更明智的决策，将分散在各品牌官网上的规格数据集中到一个可对比的界面。

---

## 二、产品战略分析

### 2.1 市场定位

| 维度 | 当前状态 | 评估 |
|------|---------|------|
| 目标用户 | 中国农机购买者（个体农户、合作社、农场主） | ✅ 清晰 |
| 使用场景 | 购买前的规格对比研究 | ✅ 明确 |
| 竞品差异化 | 雷达图 + 胜出算法 + 中文界面 | ⚠️ 有差异化但不够深 |
| 商业模式 | 无（纯工具） | ❌ 未定义 |
| 流量来源 | 搜索引擎（长尾关键词） | ⚠️ 有认知但未充分优化 |

### 2.2 核心竞争力评估

**优势**：
- 专为中国市场设计的中文农机对比工具（竞品多为英文通用型）
- 雷达图可视化降低了参数理解门槛
- Next.js SSR 对 SEO 友好

**劣势**：
- 数据量有限（17 款），更新依赖开发者手动修改代码
- 无 UGC/社区机制，数据增长慢
- 无用户留存机制（用完即走）
- 胜出算法在 README 中描述得很强大，但**代码中实际未实现场景加权**

### 2.3 战略建议

1. **短期（1-3 月）**：修补功能缺口、SEO 优化、提升对比体验 → 提升搜索流量转化
2. **中期（3-6 月）**：接入 TractorWatch 价格数据实现动态价格、增加品牌和型号 → 扩大内容覆盖
3. **长期（6-12 月）**：用户账号系统 + 收藏 + 评论 → 从工具转向社区平台

## 三、用户体验分析

### 3.1 用户旅程地图

```
搜索 "约翰迪尔5075E对比久保田M7060"
  → 点击搜索结果
    → 着陆对比页（两个拖拉机已选中）✅ 当前支持 URL 参数
    → 查看雷达图
    → 滚动查看参数表
    → 看优势统计
    → 离开页面 ❌ 无下一步行动引导
```

### 3.2 关键 UX 问题

#### 🔴 严重问题

**1. README 中的胜出算法未实装**
- README 描述：按使用场景加权（农田作业 PTO×3 + 提升力×2 + 液压×2；运输 马力×2 + 速度×2 + 重量×1）
- 实际代码：简单比较每个参数谁更大，统计"赢了几项"
- **影响**：产品的核心差异化功能名存实亡，用户看到的"优势统计"没有场景适配

**2. TractorCard 点击后的死胡同体验**
- 卡片点击跳转到 `/compare?left=${tractor.id}`（只有左侧选中）
- 用户着陆后看到一个半空的对比页，需要自己再选右侧
- **影响**：从首页到实际对比的转化路径断裂，drop-off 率高

**3. 对比页空状态无引导**
- 初始状态只显示 "👆 请分别选择两台拖拉机开始对比"
- 没有热门对比推荐，没有搜索引导，没有品牌快捷选择
- **影响**：用户进入 `/compare` 无参数时不知道从何下手

#### 🟡 中等问题

**4. 无可分享的对比链接**
- URL 已支持参数化（`?left=xxx&right=xxx`），但没有"复制链接"按钮
- 无社交媒体分享
- **影响**：错过了自然传播机会。农机购买是群体决策场景（农户之间互相推荐）

**5. 无对比历史**
- 用户对比完离开后无法找回之前的对比
- 无"最近对比"列表
- **影响**：用户在购买决策周期内（通常数周）需要反复回到同一个对比

**6. 空状态/无结果状态的处理不充分**
- 首页搜索无结果时只显示"没有找到匹配的拖拉机"
- 没有建议"试试搜索品牌名"、"查看所有 John Deere 型号"

#### 🟢 轻微问题

**7. 深色模式存在闪烁风险**
- `layout.tsx` 的内联 script 和 `Header.tsx` 的 `useEffect` 都在设置深色模式
- SSR 渲染时可能出现短暂的颜色闪烁

## 四、数据与内容策略

### 4.1 数据模型评估

当前 `Tractor` 接口设计合理（22 个字段），与 TractorVIN 对齐。但存在以下问题：

| 字段 | 问题 |
|------|------|
| `priceRange` | 静态字符串（"¥28-35万"），无结构化价格数据，无法参与算法计算 |
| `yearIntroduced` | 有数据但从未展示在 UI 上 |
| `fuelType` | 全部是"柴油"，无差异化价值 |
| `transmissionType` | 文本描述，无法结构化和对比（如"几前进档/几后退档"） |
| `tireSize` | 有数据但未展示在参数表中 |

### 4.2 数据增长策略

- **问题**：17 款型号全由开发者手动录入，无扩展机制
- **建议**：
  1. 建立数据贡献模板（Google Form → PR 工作流）
  2. 从 TractorVIN 解码库反向生成更多型号数据
  3. 接入 TractorWatch 动态价格数据替换静态 priceRange

## 五、增长与 SEO

### 5.1 当前 SEO 状态

| 检查项 | 状态 | 说明 |
|--------|------|------|
| `<title>` | ⚠️ | 全局固定标题，对比页无动态标题 |
| `<meta description>` | ⚠️ | 同上，对比页无动态描述 |
| Open Graph | ❌ | 无 og:title / og:image |
|结构化数据 | ❌ | 无 JSON-LD Product schema |
| Sitemap | ❌ | 无 sitemap.xml |
| Canonical URL | ❌ | 无 |
| URL 结构 | ✅ | `/compare?left=xxx&right=xxx` 参数化良好 |

### 5.2 SEO 机会分析

目标关键词示例：
- "约翰迪尔 5075E 对比 久保田 M7060"（长尾，高意图）
- "拖拉机参数对比"（中频）
- "100马力拖拉机哪个好"（高流量，信息型）
- "John Deere vs Kubota 拖拉机"（中英文混合）

**预估**：仅对比页动态 title 优化即可提升 CTR 15-25%。

### 5.3 增长飞轮设计

```
优质对比内容 → 搜索引擎收录 → 用户搜索发现 → 
使用对比工具 → 分享给朋友/社群 → 更多搜索流量 → 
更多品牌关注 → 更多数据收录 → 更优质的内容
```

**当前断裂点**：分享环节缺失（无可分享功能），数据增长环节缺失（无新数据收录机制）

## 六、技术架构评估

### 6.1 当前架构

```
Next.js 16 App Router
├── SSR (服务端渲染) → SEO 友好
├── Client Components (交互) → useSearchParams, useState
├── 静态数据 (TypeScript 数组) → 零 API 延迟
└── Tailwind CSS 4 + Recharts → 快速 UI 开发
```

### 6.2 技术债务

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 胜出算法未实装 | 🔴 高 | 产品核心功能缺失 |
| `.next/` 目录被提交 | 🟡 中 | 应加入 .gitignore |
| 无测试 | 🟡 中 | 计算逻辑无单元测试 |
| 无错误边界 | 🟢 低 | 组件崩溃无兜底 |
| `compareFields` 在 data 文件中 | 🟢 低 | UI 配置放在数据文件中，应拆分 |

## 九、总结

TractorCompare 是一个**方向正确、基础扎实但执行不完整**的产品。它在产品设计上有很好的想法（场景化胜出算法、生态系统联动），但最关键的几个差异化功能只停留在 README 描述中，未在代码中实现。

**本次优化的核心逻辑**：
1. 把"说的"变成"做的"——实现场景加权胜出算法
2. 让"来了的人"能"找得到"——SEO 动态元数据
3. 让"用完的人"能"回得来"——对比历史 + 分享
